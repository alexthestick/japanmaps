# Lost in Transit Web — Production Audit
**Date:** 2026-07-03 · **Scope:** full codebase + **live database** (RLS policies, security advisors, and indexes verified against the actual Supabase project, not inferred from code)

---

## 1. Executive Summary

The application architecture is fundamentally sound — route-level code splitting, a disciplined single-fetch store cache, tiered map markers, real prerendering for SEO, and a CLAUDE.md culture that has demonstrably prevented regression relapses. The product thinking (Radar, quests, finds, passport) is ahead of the engineering hygiene around it. **The biggest risk is not performance or design — it's that the database will let any person who creates a free account edit or delete all 899 stores**: the live `stores` table carries leftover permissive RLS policies (`auth.role() = 'authenticated'` for UPDATE/DELETE/INSERT) alongside the admin-only ones, and permissive policies OR together. A second live leak compounds it: `.env.local.bak` is committed to the GitHub remote with working Google Places and Gemini API keys. Moderation is also bypassable — users can INSERT or UPDATE their own `field_notes` to `status='approved'` because no policy constrains the status column. The biggest opportunity is cheap: the app is flying completely blind (zero analytics, zero error tracking) while sitting on unusually rich behavioral primitives (checkins, saves, finds) that could power both product decisions and user-facing features. Fix the four security items this week; they are hours of work, not days.

## 2. Prioritized Findings

### 2.1 Security 🔴 (the section that matters most)

**S1 — Any authenticated user can UPDATE / DELETE / INSERT stores. Severity: Critical · Effort: S**
Evidence (live `pg_policies`): table `public.stores` has BOTH `"Admins can update stores" (is_user_admin())` AND `"Authenticated users can update stores" (auth.role() = 'authenticated')` — same duplication for INSERT and DELETE. Permissive policies OR together, so the admin check is decorative. Anyone who signs up on `/signup` can `PATCH /rest/v1/stores` or delete the entire dataset with the anon key from the JS bundle plus their session token. Same pattern on `blog_posts`: `"Only authenticated users can manage blog posts"` (cmd=ALL, `auth.role()='authenticated'`) sits alongside the admin policies.
**Fix:** `DROP POLICY "Authenticated users can update stores" ON stores;` (and the INSERT/DELETE twins, and the blog_posts ALL policy). Keep only `is_user_admin()` policies. Then take a DB backup and check `stores.updated_at` for rows modified by non-admin accounts.

**S2 — Moderation bypass on finds. Severity: Critical · Effort: S**
Evidence: `field_notes_insert_own` has `WITH CHECK (user_id = auth.uid())` — nothing constrains `status`. `field_notes_update_own` has `USING (user_id = auth.uid())` and **no WITH CHECK at all**. The client politely inserts `status: 'pending'` (`src/pages/FindsPage.tsx:360`), but any user can insert or update their own rows to `status='approved'` via the REST API directly — self-approving UGC and defeating the entire admin review queue.
**Fix:** `WITH CHECK (user_id = auth.uid() AND status = 'pending')` on INSERT; on UPDATE add `WITH CHECK (user_id = auth.uid() AND status = 'pending')` so users can edit only pending posts (matching what the iOS `UserFindCard` UI already assumes).

**S3 — Live Google API keys committed to GitHub. Severity: Critical · Effort: S**
Evidence: `.env.local.bak` is tracked (`git ls-files`), present in the initial commits (`dd632e0`, `cc687ca`) and pushed to `github.com/alexthestick/japanmaps`. It contains real-looking `VITE_GOOGLE_PLACES_API_KEY=AIzaSyDbYx…` and `VITE_GOOGLE_GEMINI_API_KEY=AIzaSyBjVW…`. Removing the file now does NOT remove it from history.
**Fix (in order):** 1) Rotate both keys in Google Cloud Console *today* and add API restrictions (Places API only; HTTP referrer lock for any browser key). 2) `git rm --cached .env.local.bak` + commit. 3) Optionally purge history (`git filter-repo`) — but rotation is the real fix; treat the old keys as burned. 4) Check Google Cloud billing for anomalous usage. Also: root `.env` holds a live `ANTHROPIC_API_KEY` and `SUPABASE_SERVICE_ROLE_KEY` on disk — verified never committed (gitignored), but a service-role key in a folder that gets zipped/shared/screen-shared is one accident from being S1. Move server-side secrets out of the repo root (Vercel env vars) and rotate the service-role key.

**S4 — Anonymous, unlimited INSERT into `store_suggestions`. Severity: High · Effort: M**
Evidence: live advisor `rls_policy_always_true` — `suggestions_insert` is `WITH CHECK (true)` for `anon, authenticated`. One `curl` loop fills your admin queue with a million rows.
**Fix:** move the insert behind an Edge Function with per-IP throttling (or at minimum a length/shape CHECK constraint + a captcha field), revoke direct anon INSERT.

**S5 — Public buckets allow listing. Severity: Medium · Effort: S**
Evidence: advisors flag `finds-photos` and `profile-photos` — broad SELECT policies on `storage.objects` let any client *enumerate every file* (every user's photo) rather than just fetch known URLs. Public buckets don't need a SELECT policy for URL access.
**Fix:** drop `finds_photos_public_read` / `profile_photos_public_read` policies; public-bucket URL access keeps working.

**S6 — Counter RPCs executable by `anon`. Severity: Medium · Effort: S**
Evidence: advisors list `increment_visit_count`, `decrement_visit_count`, `_increment/_decrement_store_checkin_count`, `handle_new_user` as SECURITY DEFINER functions executable by `anon` via `/rest/v1/rpc/…`. The trigger-style ones may error without trigger context, but exposure is free to remove: `REVOKE EXECUTE ... FROM anon, authenticated;` for anything not meant as public API. While there: the `spatial_ref_sys` REVOKE documented in CLAUDE.md **was never actually run** — the advisor still fires. Run it.

**S7 — Leaked-password protection disabled. Severity: Medium · Effort: S**
The web app has email/password auth (`LoginPage`/`SignupPage`) and Supabase's HaveIBeenPwned check is off. One toggle in Auth settings.

*Fine as-is:* `gps_checkins` has no client INSERT policy (Edge-Function-only writes — correct); `saved_stores`/`reports`/`find_comments` policies are properly self-scoped; `KURB_API_KEY` posture matches the guardrail; `verify_jwt:false` on `get-kurb-inventory` is acceptable since it's a read-only proxy keyed by vendor_id — the only tightening worth doing is validating `vendor_id` is an integer and rate-limiting.

### 2.2 Data layer / Supabase

**D1 — Duplicate/overlapping RLS policies everywhere. Severity: Medium · Effort: S** — `saved_stores` has 7 policies (3 SELECT variants, 2 INSERT, 2 DELETE), `profiles` has 2 identical SELECTs, `field_notes` 2 SELECTs. Beyond confusion, every duplicate permissive policy is evaluated per row per query (this is what most of the 70KB performance-advisor output is about). Consolidate to one policy per table/command.
**D2 — Index coverage: genuinely good.** `stores_slug_idx` unique ✓, GIN on categories ✓, GIST on location ✓, `field_notes(store_id/status/user_id/created_at)` ✓, unique `(user_id, store_id)` on both checkins and saves ✓. One gap: `field_notes` store-page queries filter `store_id + status + created_at desc` — a composite `(store_id, status, created_at DESC)` would replace three single-column lookups, but at current row counts this is theoretical. No action needed this quarter.
**D3 — CLAUDE.md staleness: the `get_stores_with_coordinates` slug note is obsolete.** The (uncommitted!) `src/hooks/useStores.ts:14` now calls `get_all_stores_json`. Update CLAUDE.md; see A2 about committing this.

### 2.3 Architecture & code quality

**A1 — `HomePage.tsx` is a 1,572-line god component. Severity: High · Effort: L** — Browse + list + spotlight + full Radar mode (session refs, quest state, exit confirmation, haul prompts) in one file with ~30 state atoms. Same disease as the iOS map screen. Extract a `RadarProvider`/`useRadarSession` module; HomePage should compose modes, not implement them. This is where your next regression will come from — the file already needs its own comment-navigation system.
**A2 — 2,165 lines of uncommitted changes across 12 files. Severity: High · Effort: S** — `git status` shows a half-finished admin overhaul (`AdminDashboard.tsx` ±956 lines, new `DataQualityTab/DuplicateDetector/NeighborhoodFixer/OverviewTab`) **plus** the `useStores` RPC swap sitting untracked. One `git checkout .` or disk failure erases weeks of work; worse, the deployed site may be running different data-fetching code than your working tree. Commit the RPC swap separately today; put the admin work on a branch.
**A3 — 100 `any` escapes, 111 raw `console.*` calls. Severity: Medium · Effort: M** — `tsconfig` is `strict: true` (good) but `as any` clusters in exactly the risky spots (Supabase rows, map events). The `logger` util exists but 111 call sites bypass it, so production consoles are noisy and `App.tsx:7` even logs a build version banner. Sweep `console.*` → `logger`, then chip at `any`s in `types/store.ts` boundaries.
**A4 — Duplicated store-detail implementations. Severity: Medium · Effort: L** — `StoreDetail.tsx` (desktop panel), `MobileStoreDetail.tsx`, `BottomSheetStoreDetail.tsx`, and `StoreDetailPage.tsx` each reimplement hours/links/stats/gallery rendering with drift (e.g., different haul-count logic, different gallery presets). Extract shared `<StoreFacts/>`, `<StoreGallery/>`, `<StoreLinks/>` primitives; keep layout shells separate.
**A5 — `@anthropic-ai/sdk`, `axios`, `sharp` misfiled.** `@anthropic-ai/sdk` is in `dependencies` but only used by `scripts/generate-store-descriptions.mjs` — move to devDependencies (it's not bundled since unimported in `src`, but the manifest lies about the runtime). Both `html2canvas` AND `html-to-image` ship for the same job (radar share cards vs admin carousel) — pick one, drop ~150KB from installs.

### 2.4 Performance

**P1 — Largely healthy, and the hard problems are already solved.** Route-level `lazy()` for all 24 pages (`App.tsx:15-38`), Mapbox GL confined to the `/map` chunk and lazy `StoreMiniMap` on detail pages, tiered markers with viewport culling + 80–200 marker caps (`MapView.tsx:791-801`), memoized distance sorting on a 100m snap grid, 250ms viewport throttle. This is above-average work. Remaining items are moderate:
**P2 — Marker rendering is still DOM-based. Severity: Medium · Effort: L** — Up to 200 React `<Marker>` components re-render on tier changes; `supercluster` is in package.json but the render path maps raw stores. At 899 stores it holds; at 2,000+ it won't. Long-term: move dots to a GeoJSON source + circle/symbol layers (the iOS app already proves this architecture), keep DOM markers only for spotlight/radar tiers.
**P3 — `ikUrl` preset rule violated in the desktop detail panel. Severity: Low-Medium · Effort: S** — CLAUDE.md: "never use `thumb` on desktop cards — blurry on Retina." `StoreDetail.tsx:219` renders the desktop gallery grid at `'thumb'`; `BottomSheetStoreDetail.tsx:214` gallery likewise. Both are the largest images in their panel. Switch to `'card'`. (`MobileListView.tsx:326` at `thumb` is defensible for small cells but worth an eyeball on a 3x device.)
**P4 — Radar share cards ship 2 screenshot libraries** (see A5). Also `lenis` smooth-scroll runs globally on the landing page — verify it's not mounted on `/map` where it fights Mapbox gestures.

### 2.5 Design & UX

**U1 — The aesthetic is real but hand-maintained. Severity: Medium · Effort: M** — There is no token system; the neon language lives in hundreds of hardcoded values (`#22D9EE` appears as `#22d9ee`, `rgba(34,217,238,…)` at 6 different alphas, `cyan-300/400/500` interchangeably). The iOS app has `theme.ts`; the web has vibes. Result: drift is visible — some cyan glows are `0 0 30px`, others `0 0 15px`, at least three different border-alpha conventions on cards. Extract Tailwind theme extensions (`colors.neon.cyan`, shadow presets) and sweep.
**U2 — Mobile two-tap markers are undiscoverable. Severity: Medium · Effort: M** — First tap shows a label, second opens the sheet (`HomePage.tsx:143-159`), with a 3s reset. No affordance tells users this; testers will report "sometimes tapping does nothing." The iOS app went single-tap → sheet and it's better. Recommend: single tap opens the sheet at a low peek snap (the label preview's job is served by the sheet header itself).
**U3 — Spotlight mode is the best feature nobody finds. Severity: Medium · Effort: S** — It hides behind a "Search this area" button that appears only at zoom ≥14 after panning, and the name promises search, not curation. Rename the CTA ("✦ Pick 5 for me"), and surface it once via the same one-time tooltip pattern the Radar pill already has (`HomePage.tsx:103-113`).
**U4 — First-run experience is one 4-second tooltip.** A first-time visitor lands on `/map` with 899 dots, no legend on mobile, no "what is this." A single dismissible 3-step overlay (Browse → Spotlight → Radar) would fix onboarding for both features above. The `RadarOnboardingCard` pattern already exists to copy.
**U5 — Empty/error states are inconsistent.** FindsPage has skeleton + fade polish; HomePage's error state is a bare centered `<h2>` with no retry; several admin views render nothing while loading. Standardize on the FindsPage pattern. Motion quality is otherwise genuinely good — direction-aware store-page slides (`App.tsx:50-57`) are a nice touch most apps skip.

### 2.6 Accessibility — Medium severity overall, Effort: M

50 `aria-label`s across 217 files is better than most indie apps, but: map markers are `<div onClick>` (`MapView.tsx:91-130` AtmosphereDot/DiscoveryDot) — no `role="button"`, no `tabIndex`, invisible to keyboards and screen readers; the two-tap pattern (U2) is entirely inoperable non-visually; store labels are `pointer-events-none` divs. Bottom sheets: verify focus trap + `Escape` handling in `BottomSheet.tsx` and that background content gets `aria-hidden`. Contrast: `text-gray-400 on gray-900` body copy passes AA large but the `text-white/30` timestamps (`StoreFindsSection.tsx:88`) and `text-gray-600`-on-black bits fail. Pragmatic path: fix keyboard access on the list view + filters + sheets first (the map is a hard a11y problem; the list is your accessible equivalent — make it excellent), add `role/tabIndex/onKeyDown` to markers second.

### 2.7 SEO — genuinely strong, two gaps

The stack is above-average: real prerendering (`build:seo` → puppeteer over all routes, priority-only on Vercel), DB-driven sitemap that refuses fallback slugs (`scripts/generate-sitemap.mjs`), per-page `SEOHead` with store + breadcrumb JSON-LD, category-aware meta descriptions (`StoreDetailPage.tsx:286-347`), UUID→slug 301-style redirect. Gaps: **(1)** `index.html` has no `<link rel="canonical">` and `og:url` hardcodes `/` — fine for the shell, but confirm `SEOHead` emits canonical on every prerendered route, *especially* the UUID variant pointing at the slug URL (otherwise duplicate-content risk on legacy links). **(2)** The prerender pipeline is load-bearing and completely untested — one Puppeteer/Chrome-version hiccup on Vercel silently ships an unprerendered build. Add a post-build assertion (grep 3 known store pages' HTML for their `<title>`) that fails the deploy.

### 2.8 Analytics & observability

**Confirmed: nothing.** No PostHog/GA/Plausible/Sentry anywhere in `package.json` or `src`. You cannot answer "do people use spotlight?", "what % of mobile users survive two-tap?", or "did the last deploy throw errors?" — while sitting on a product whose entire roadmap depends on those answers. **Severity: High · Effort: S.** Concrete fix: PostHog (JS snippet, EU or US cloud, free tier covers you) for product events + session replay, `@sentry/react` with the Vite plugin for errors + source maps. One afternoon. Instrument exactly six events to start: `map_loaded`, `store_opened`, `spotlight_activated`, `radar_started`, `checkin_success`, `find_submitted`.

### 2.9 Repo hygiene

39 markdown files, 22 `vite.config.ts.timestamp-*.mjs` artifacts, 4 Python tools + `__pycache__`, 5 CSVs of store data, a 1.7MB PNG, and mystery `Folder/` + `Maps/` dirs — all at root. **Severity: Low (with one Medium) · Effort: S.**
- **Medium sub-finding:** the CSVs (`japan stores ready for sql file.csv`, 237KB; `Japan_Clothing_Stores_Clean.csv`) are your proprietary dataset sitting in a repo that also had leaked keys — check whether they're tracked (`git ls-files | grep csv`); if the repo ever goes public or is shared, that's the entire moat in a download.
- Move: docs → `/docs/archive/` (keep CLAUDE.md + README at root), Python carousel tools → `/tools/carousel/` (they're load-bearing for your Instagram workflow — `carousel_app.py` is 91KB of real tool — but they don't belong beside `index.html`), CSVs → `/data/` (gitignored), delete timestamp artifacts (already gitignored, just `rm`), delete `.env.local.bak` after S3.
- `.gitignore` blanket-ignores `*.py` — which means the carousel tools are *untracked and unbacked-up*. If they matter, un-ignore them deliberately.
